This was a high school capstone project. Looking back, it's a nice example of first principles thinking about system design.

The problem was straightforward: build a system that lets people book train tickets. Sounds simple until you think about it. How do you handle two people trying to book the last available seat simultaneously? How do you prevent overbooking? What happens when a reservation needs to be cancelled?

I built a Python backend with MySQL database handling passenger details, routes, availability checks, and ticket generation. The key was thinking about the data structure first. A normalized schema with proper relationships between passengers, trains, routes, and bookings. Validation logic to ensure consistency. Printable tickets.

This was my first real exposure to databases. I learned that schema design matters. If you design the tables right, the queries are simple. If you design them wrong, nothing works. I learned about transactions—what it means to atomically book a seat and reduce availability in one operation, not two.

Did I implement perfect concurrent access patterns? No. The requirements were modest. But the way I thought about the problem was right: data first, then logic, then UI. That's the foundation of good system design.

## What It Does

**Core Functionality:**
- **Passenger Management**: Capture and validate passenger details (name, age, ID, contact)
- **Route Management**: Train routes with departure/arrival times and station information
- **Availability Checking**: Real-time seat inventory tracking across fare classes
- **Reservation Engine**: Book seats, generate unique PNR (reservation number), handle cancellations
- **Ticket Generation**: Print-ready tickets with passenger info, PNR, and journey details

## Technical Architecture

**Database Layer (MySQL):**
- Normalized schema with separate tables for passengers, trains, routes, and bookings
- Foreign key constraints ensuring data integrity
- Indexed searches on PNR and passenger ID for quick ticket retrieval

**Backend (Python):**
- Object-oriented design with separate classes for Passenger, Train, and Booking entities
- Validation logic ensuring only valid bookings are accepted
- Transaction handling for atomicity during reservation

**Frontend (HTML/CSS):**
- Simple, functional interface for entry and ticket printing
- Form validation before submission
- Clear ticket format optimized for printing

## Key Learnings

This early project reinforced:
1. **Schema design matters**—a clean, normalized structure made queries simple and data consistent
2. **User workflows are part of the product**—the ticket layout and booking flow needed to match real-world expectations
3. **Validation prevents problems**—checking age for children, preventing overbooking, etc. prevents downstream chaos

While simple by today's standards, this project taught me to think like a systems builder, not just a coder.
