Grocery shopping is surprisingly inefficient. You don't know which store actually has better prices on what you're buying, can't easily compare total basket costs, and tracking items manually is tedious. We built a system to solve that.

## System Architecture

**Data Pipeline**
Scheduled scraper runs hourly against 100+ retail websites. Extracts product name, price, store, category using CSS selectors and regex patterns. Data normalized and inserted into PostgreSQL in batches (1000-item chunks) for throughput.

Anomaly detection runs post-insert: z-score outlier detection flags prices >2σ from rolling 30-day average. Flagged prices trigger user alerts if item is on watchlist. Detection happens async—scrapes complete immediately, analysis happens in background.

**Database: PostgreSQL**

Schema designed for actual query patterns:

```sql
-- Core tables
products(id, sku, name, category, standard_unit)
stores(id, name, location, chain)
prices(id, product_id, store_id, price, scraped_at)
watchlist(id, user_id, product_id, alert_threshold)

-- Indexes that matter
CREATE INDEX idx_prices_product_store_date 
  ON prices(product_id, store_id, scraped_at DESC);

CREATE INDEX idx_prices_category_store 
  ON prices(category_id, store_id) 
  INCLUDE (price);

-- Materialized views for fast aggregations
CREATE MATERIALIZED VIEW avg_price_by_product AS
  SELECT product_id, 
         AVG(price) as avg_price,
         MIN(price) as min_price,
         STDDEV(price) as stddev_price
  FROM prices 
  WHERE scraped_at > NOW() - INTERVAL '30 days'
  GROUP BY product_id;
```

This design reduced query latency from 5+ seconds to under 200ms.

**Backend: Node.js + Express**

Endpoint structure:
- `GET /products/search?q=...` – full-text search using PostgreSQL trigram index
- `POST /baskets/compare` – takes array of items, returns cost breakdown by store
- `GET /prices/:productId/history` – price timeseries for visualization
- `POST /alerts/watch` – create watchlist entry with threshold alert

Connection pooling with `pg` library (20 connections). Query results cached with Redis (TTL: 10 min for product data, 1 hour for aggregations). Sensitive queries wrapped in prepared statements to prevent SQL injection.

Request validation with Joi. Error middleware sanitizes responses—no SQL error details sent to client.

**Frontend: React + TypeScript**

Components:
- Product search with debounced API calls (300ms wait before requesting)
- Basket comparison showing itemized breakdown, total, and best option highlighted
- Price history chart using Recharts, 30-day view with trend overlay
- Watchlist management with threshold alerts

State management through Context API. Real-time price updates trigger re-render through WebSocket connection (if user has page open during scrape run).

**Performance Optimization**

Basket comparison query was initially slow: for each product and store combination, calculate total cost. Refactored to:
- Precompute store totals in background (updated after each scrape)
- Query against precomputed table instead of aggregating on-demand
- Result: basket comparison dropped from 8s → 150ms

Pagination on product search and price history. Lazy load price charts only when tab is viewed.

**Results**

System handles 50K+ products, 100+ retailers, 2M+ price records. Daily users get alerts for their watchlisted items. Basket comparison is frequently used feature—tells users exactly where to shop.
